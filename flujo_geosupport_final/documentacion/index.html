Flujo final Geosupport para imagenes drone PAO

Proceso consolidado para normalizar imagenes, preparar rutas de datastore, cargar al mosaic dataset, actualizar atributos criticos y generar el MAPX de publicacion con rasters directos referidos al datastore.

Resumen del flujo

1. NormalizacionPreparar imagenes tecnicamente consistentes antes de cargarlas.Salida: GeoTIFF RGB limpio en normalizadas.
2. Preparacion de pathsResolver nombre oficial y ubicacion en datastore.Salida: manifiesto 04_ready_for_datastore.csv.
3. Carga al mosaicoCopiar, registrar, construir footprint y completar atributos.Salida: registros validos en mosaic dataset.
4. Footprints indiceExportar geometrias del mosaico a la GDB de indice y recalcular nombres.Salida: capa de footprints sincronizada.
5. MAPXCrear el mapa publicable desde los paths raster registrados en el mosaico.Salida: MAPX ordenado con TIFF directos.
El flujo final esta pensado para trabajar con imagenes normalizadas antes de entrar al calculo de paths y a la carga al datastore. La carpeta normalizadas funciona como area de trabajo confiable; el nombre oficial de cada TIFF se define en la etapa de preparacion de paths. Las rutas y parametros principales se controlan desde flujo_geosupport_final/settings.json.

Logica general del proceso

Este flujo no es una carga generica de rasters. Es un proceso personalizado para PAO, porque el nombre final, el directorio fisico, los atributos del mosaic dataset y el mapa de publicacion dependen de reglas de negocio y de datos espaciales.

1

Normalizar primero para estabilizar la imagen

Antes de copiar al datastore se corrigen los problemas tecnicos detectados en la entrega: bordes blancos o negros, imagenes colormap y lecturas inconsistentes de bandas. La salida queda lista para ArcGIS como RGB real de tres bandas.

Entrada
TIFF originales del cliente.
Regla
Rasterio streaming y salida RGB.
Control
Revisar CSV de normalizacion.
2

Calcular nombre y destino con datos confiables

El nombre final no se toma directamente del texto original. La fecha viene primero de la tabla externa de fechas (fechas.json o Excel) y, si no existe alli, se extrae por regex desde el nombre; el sector viene del cruce espacial. Con ambos datos se arma el nombre oficial y se determina la carpeta destino.

Entrada
Folder normalizadas.
Regla
fecha + sector.
Control
Solo pasan registros con path destino completo.
3

Cargar al datastore y consolidar en el mosaic dataset

El manifiesto aprobado es la fuente de verdad para copiar los archivos. Luego se agregan al mosaico, se construyen footprints y se actualizan campos criticos para que el visor, las consultas y las descargas funcionen de forma consistente.

Entrada
04_ready_for_datastore.csv.
Regla
Copiar, agregar raster, construir footprint, actualizar atributos.
Control
Validar resultados de carga y atributos.
4

Sincronizar footprints en la GDB de indice

Despues de cargar al mosaico, se exportan las geometrias footprint de las imagenes nuevas, se ajusta el mapeo de campos y se agregan a la capa historica de footprints en la GDB. Luego se recalcula Nombre_de_Vuelo con la numeracion esperada y se actualiza ese dato en el mosaic dataset.

Entrada
Registros cargados correctamente.
Regla
Exportar footprint, mapear campos, append y recalcular.
Control
CSV de nombres agregados y actualizaciones.
5

Publicar rasters directos usando el mosaico como inventario

El MAPX se construye agregando los TIFF directos desde los paths registrados en el mosaic dataset. El mosaico se usa para identificar que imagenes existen, sus fechas y su path fisico; la capa final queda referida al archivo raster del datastore.

Entrada
Mosaic dataset PAO + paths exportados.
Regla
Una capa raster directa por RasterPath.
Control
MAPX con grupo Imagenes Drone ordenado.

Entradas operacionales

Carpeta de imagenes originales

Es la carpeta de entrega del cliente. El flujo la recorre de forma recursiva y genera una subcarpeta normalizadas con los GeoTIFF que se usaran para el resto del proceso.

Fechas

La fecha de adquisicion se resuelve primero desde la tabla externa de fechas (json/fechas.json o Excel). Si el archivo no aparece alli, se intenta extraer desde el nombre del archivo usando patrones regex. Esa fecha participa directamente en el nombre de salida: CL_MLP_PAO_IF_Ortho_yy_mm_dd_sector.tif.

Feature class de sectores

El sector no se infiere desde el nombre original. Se obtiene por cruce espacial contra la capa de indices/footprints configurada en el modulo del flujo.

Mosaic dataset y datastore

El datastore recibe el TIFF final normalizado y sus archivos auxiliares asociados cuando existen. El mosaic dataset registra la imagen, construye el footprint y conserva los atributos que consumen los visores y servicios.

Etapa 1: normalizacion RGB

La normalizacion se ejecuta antes de preparar rutas. El objetivo es que todas las imagenes que entran al datastore sean GeoTIFF RGB de tres bandas, comprimidos, sin depender de una mascara de footprint externa.

Esta etapa se separo del resto porque fue el punto mas sensible del proceso. Las imagenes llegaban con distintas condiciones internas: algunas con fondo visible, otras como colormap, otras con archivos auxiliares que influian en la visualizacion. Por eso el flujo primero genera una version tecnicamente estable y despues continua con la logica GIS.

  • Entrada: TIFF originales de la entrega.
  • Salida: carpeta normalizadas dentro de la carpeta de entrada.
  • Nombre de salida: se conserva como nombre operativo temporal. El nombre oficial se calcula despues con fecha y sector.
  • Metodo: lectura por ventanas con rasterio, conversion a RGB y limpieza de fondo por valores dominantes/bordes.
  • Compresion: GeoTIFF optimizado, sin perdida de resolucion.

Problemas encontrados y correccion aplicada

Problema observadoCausa practicaDecision del flujo final
Bordes negros o blancos visibles fuera de la imagen util.Algunas imagenes traian valores de fondo en los bordes o archivos auxiliares/overviews que ArcGIS podia seguir mostrando.Normalizar con rasterio streaming, detectar el fondo desde bordes/valores dominantes y generar un nuevo TIFF comprimido antes de cargar al datastore.
Imagenes colormap.Se encontraron rasters definidos con colormap. En pantalla podian verse como imagen a color, pero internamente no eran RGB real de tres bandas.Convertir la salida a GeoTIFF RGB explicito de 3 bandas, con bandas 1, 2 y 3 listas para Red, Green y Blue.
Imagenes que se veian diferente entre mosaic dataset y TIFF directo.El mosaico podia aplicar su propia lectura/simbologia, mientras el TIFF directo exponia overviews o valores de fondo.La normalizacion deja el TIFF directo estable para que pueda ser agregado al MAPX desde su path del datastore.
Procesamiento lento o consumo alto de memoria.Algunos rasters son grandes y no conviene cargarlos completos.El script procesa por ventanas o bloques, manteniendo la resolucion y reduciendo consumo de memoria.
Esta etapa no reemplaza directamente los archivos del datastore. Primero produce copias normalizadas y el resto del flujo usa esas copias como entrada confiable.

Etapa 2: preparacion de paths datastore

El notebook lee la carpeta normalizadas y calcula el destino final de cada imagen. El nombre final se forma con dos componentes principales: fecha de adquisicion y sector espacial.

De donde vienen los nombres

El nombre final no depende del texto descriptivo original de la imagen. El flujo arma el nombre oficial con una estructura fija que permite ordenar, publicar y comparar contra el mosaic dataset:

CL_MLP_PAO_IF_Ortho_yy_mm_dd_sector.tif

  • yy_mm_dd viene primero de json/fechas.json o del Excel de fechas. Si el archivo no esta en esas tablas, se calcula con regex sobre el nombre del archivo.
  • sector viene del cruce espacial de la imagen contra la capa de indices/sectores.
  • Si el nombre resultante se repite, se agrega una secuencia para evitar colisiones.

De donde vienen los directorios destino

El directorio destino se resuelve despues de conocer el sector. Cada sector se mapea a una carpeta operacional del datastore, por ejemplo Chacay_El_Mauro_Drone, El_Mauro_Drone, Chacay_Drone o Puerto_Punta_Chungo_Drone. La ruta final combina:

datastore raiz / carpeta operacional / yy_mm / nombre oficial.tif

El folder mensual yy_mm se obtiene de los primeros cinco caracteres del token de fecha. Por ejemplo, una fecha 26_05_10 cae en el folder 26_05.

Etapa 3: carga al datastore y mosaic dataset

Con el manifiesto ya definido, el flujo copia cada TIFF normalizado al path destino y luego lo registra en el mosaic dataset. La copia incluye el TIFF y los archivos auxiliares asociados generados o validados previamente por ArcGIS Pro, por ejemplo .tif.ovr, .tif.aux.xml y XML laterales cuando existan. Esta etapa tambien construye footprints y actualiza los campos criticos de negocio.

Que usa como entrada

La entrada de esta etapa es el manifiesto validado de la etapa 2. En ese manifiesto ya existe una relacion explicita entre:

  • Path de la imagen normalizada de trabajo.
  • Path destino definitivo en el datastore.
  • Nombre oficial Name que debe existir en el mosaic dataset.
  • Sector y fecha de adquisicion que se usaran para atributos.

Despues de agregar el raster y construir su footprint, el flujo abre el registro del mosaic dataset por Name y actualiza los campos criticos con arcpy.da.UpdateCursor. Esta actualizacion es parte de la carga, no una revision posterior.

Etapa 4: actualizacion de footprints en GDB de indice

Despues de cargar y construir los footprints en el mosaic dataset, el flujo replica esas geometrias hacia la capa de footprints/indice mantenida en la GDB corporativa. Esta parte existe para mantener sincronizada la capa historica que consume el visor y que contiene campos con nombres distintos a los del export del mosaico.

Normalizacion adicional de datos

La capa de footprints no solo recibe geometria. Tambien se recalcula Nombre_de_Vuelo para mantener la convencion historica:

numero_secuencial_yy_mm_dd_sector

El numero se asigna recorriendo la capa ordenada por Name ASC; la fecha sale de Fecha_Adqu y el sector de Sector. Luego ese valor se devuelve al campo NombreVuelo del mosaic dataset para que ambos repositorios queden alineados.

Etapa 5: MAPX desde paths raster del mosaic dataset

El MAPX se genera leyendo el mosaic dataset completo como inventario y exportando sus paths raster con ExportMosaicDatasetPaths. Luego cada imagen se agrega al mapa como TIFF directo, referido al path fisico del datastore.

Por que se usa el mosaic dataset en esta etapa

El mosaic dataset se usa como inventario de carga, no como capa final de visualizacion en el MAPX. Desde alli se obtiene que imagenes estan cargadas, su Name, su fecha y el RasterPath real. Con esa informacion se agregan las imagenes directas al grupo Imagenes Drone y se ordenan por fecha.

  • Se excluyen los registros de overview cuyo Name comienza por ov.
  • Se exportan los paths reales de raster con ExportMosaicDatasetPaths.
  • Cada capa recibe un label limpio sin prefijo CL_MLP_PAO_IF_Ortho_ ni extension .tif.
  • Las capas se agregan como rasters directos y se insertan en Vuelos Drone PAO > Imagenes Drone.
  • El orden queda por fecha, de mas nueva a mas antigua.
  • La salida es solo MAPX; no se modifica el APRX original.

De donde vienen las capas del MAPX?De los paths raster exportados desde el mosaic dataset. Cada capa queda referida al TIFF del datastore.
Por que se consulta el mosaic dataset?Porque alli esta el inventario consolidado: Name, fecha y path raster registrado.
Como se ordenan?Por fecha, dejando las imagenes mas nuevas arriba.
Que se entrega al final?Un archivo MAPX listo para usar en el proceso de publicacion.
